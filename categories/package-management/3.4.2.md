# 3.4.2 - Index Files Are For Public API Artefacts Only

Priority | ID | Category | Criteria | Impacts
---------|----|----------|----------|--------
Minor | [3.4.2][3.4.2] | [Package Management][package-management] | Index files must only export the public API. | [Adoption][ADOPTION], [Project Maintenance][PROJECT-MAINTENANCE]

Table of Contents:
- [Description](#description)
- [Discussion](#discussion)
  - [Keep The Internal API Away From End-Users](#keep-the-internal-api-away-from-end-users)

## Description

Index files must only export the public API.

_Index files_ are used to re-export artefacts from:

* individual files in the folder
* index files in sub-folders

```typescript
export * from "./ContentType";
export * from "./MediaType";
export { InvalidMediaTypeError } from "./Errors";
```

Index files in _internal modules_ are permitted to export non-public API artefacts.

## Discussion

### Keep The Internal API Away From End-Users

Sometimes, our public API code needs to re-use functions that are for internal use only.

When that happens, we have to `export` the internal functions out of the files that they're defined in. If we don't do that, the rest of our code can't `import` and use these internal functions.

We can't stop end-users from `import`ing these internal functions directly into their code. If they want to use the full path to the individual module file, they can. What we can do, though, is at least make it as obvious as possible that an artefact isn't part of the public API.

We can support that aim by making sure that internal functions (and any other `exported` artefacts that aren't part of the public API) can't be imported from the package's top-level index file.
